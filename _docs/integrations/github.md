# GitHub App integration setup

This page documents every setting registered in
`integrations/settings_registry.py` under the `github` group. Each
section follows the same template — Purpose, Without it, Where to find
it, Prereqs, Rotation, Test vs live.

The platform reads content from the private `AI-Shipping-Labs/content`
repository via a GitHub App. The App authenticates as itself (with
the App private key signing a short-lived JWT), trades the JWT for an
installation access token, and uses that token for the rest of the
sync (`integrations/services/github_sync/client.py`).

Direct deep-link URLs are intentionally written in code blocks so they
do not render as clickable links. Copy them into the browser.

## GITHUB_APP_ID

Purpose: Numeric ID of the GitHub App, set as the `iss` claim on the
JWT generated by
`integrations/services/github_sync/client.py:generate_github_app_token`.
GitHub looks up the App by this id to validate the JWT signature.

Without it: `generate_github_app_token` raises `GitHubSyncError` with
`GitHub App credentials not configured. Set GITHUB_APP_ID,
GITHUB_APP_PRIVATE_KEY, and GITHUB_APP_INSTALLATION_ID, [...]`.
Content sync fails — no articles, courses, projects, recordings,
links, or interview questions update from the content repo.

Where to find it:

- GitHub App settings page > "About" section > "App ID" field.
- Direct link, replacing `<your-app>`:

  ```
  https://github.com/settings/apps/<your-app>
  ```

  (or under an organization: `https://github.com/organizations/<org>/settings/apps/<your-app>`)
- The App ID is a small integer (typically 6-7 digits).

Prereqs:
- A GitHub App created on the org that owns the content repo.
- The App is installed on the content repo with at least these
  repository permissions:
  - "Contents": Read-only (to clone)
  - "Metadata": Read-only (always required for any App)
  - "Webhooks": Read & write (so the App can manage push hooks)

Rotation: n/a. The App ID is permanent for the lifetime of the App.
There is no rotation.

Test vs live: n/a. The same App can serve dev and prod with different
installations; pin the `GITHUB_APP_INSTALLATION_ID` per environment
instead.

## GITHUB_APP_INSTALLATION_ID

Purpose: Installation ID of the GitHub App on the content org.
Different from `GITHUB_APP_ID` — the App is the global identity,
while each installation is the App's grant on a specific
account/org. Used by `generate_github_app_token` to mint the
installation access token at
`/app/installations/<installation_id>/access_tokens`.

Without it: Same hard failure as missing `GITHUB_APP_ID`. Without an
installation, the App has no scope on any repo.

Where to find it:

- Org admin > "Installed GitHub Apps" > pick the App > the
  installation ID appears in the URL as the trailing integer.
- Direct link, replacing `<org>`:

  ```
  https://github.com/organizations/<org>/settings/installations
  ```

- Or on a personal account:

  ```
  https://github.com/settings/installations
  ```

  Click "Configure" on the App; the URL becomes
  `https://github.com/settings/installations/<INSTALLATION_ID>`.

Prereqs:
- The App must be installed on the org or user that owns the content
  repo.
- The installation must grant access to at least the content repo
  (either "All repositories" or "Selected repositories" with
  `AI-Shipping-Labs/content` ticked).

Rotation: Installation IDs are permanent per installation. If you
uninstall and reinstall the App, GitHub issues a fresh installation
ID. Update this setting and re-run sync.

Test vs live: One installation per (org × App) pair. Use different
orgs (or different Apps) per environment, and store each
environment's installation id in the per-env override.

## GITHUB_APP_PRIVATE_KEY_SECRET_ID

Purpose: AWS Secrets Manager secret name, path, or ARN containing the
GitHub App's PEM private key. Read by
`integrations/services/github_sync/client.py:_resolve_github_app_private_key`
when the direct `GITHUB_APP_PRIVATE_KEY` setting is empty. The
recommended production path — keeps the PEM out of the Integration
Settings table.

Default: `ai-shipping-labs/github-app-private-key`. Leaving this at
its default counts as "configured" in Studio (it points at the
production secret).

Without it (and without a direct PEM): The private-key resolver
returns the empty string, `generate_github_app_token` raises
`GitHubSyncError`, content sync fails.

Where to find it:

- AWS console > Secrets Manager > "Store a new secret" > pick "Other
  type of secret" > paste the PEM content as the secret value.
- After saving, copy the secret name or ARN from the "Secret details"
  page.
- Direct link:

  ```
  https://console.aws.amazon.com/secretsmanager/listsecrets
  ```

- The value accepted here can be the friendly name
  (`ai-shipping-labs/github-app-private-key`), a relative path, or
  the full ARN
  (`arn:aws:secretsmanager:eu-west-1:<account>:secret:<name>-<hash>`).
  Boto3 resolves all three.

Prereqs:
- The IAM user defined by `AWS_ACCESS_KEY_ID` /
  `AWS_SECRET_ACCESS_KEY` must have
  `secretsmanager:GetSecretValue` on this secret.
- The PEM stored inside the secret must be the App's private key as
  downloaded from GitHub (the `BEGIN RSA PRIVATE KEY` / `END RSA
  PRIVATE KEY` block, unencrypted).

Rotation: Safe and recommended.

1. In the GitHub App settings, "Generate a private key". GitHub
   downloads a `.pem` file.
2. AWS Secrets Manager > select the secret > "Retrieve secret value"
   > "Edit" > paste the new PEM > save.
3. Web processes pick up the new PEM on the next deploy or after
   clearing the in-process cache. Worker processes always read fresh
   per call (`running_in_worker_process()` skips the cache).
4. Once the platform is using the new PEM, delete the old one from
   the GitHub App.
5. Window of impact: zero if the old PEM is still active when the new
   one is loaded — GitHub Apps support multiple active keys
   simultaneously.

Test vs live: Use a separate secret (or a separate `Secret_id`) per
environment to keep the keys isolated.

## GITHUB_APP_PRIVATE_KEY_SECRET_REGION

Purpose: AWS region where the Secrets Manager secret lives. Default
`eu-west-1`. Read by
`integrations/services/github_sync/client.py:_resolve_github_app_private_key`
and passed to `boto3.client('secretsmanager', region_name=region)`.

Without it: Defaults to `eu-west-1`. If the secret lives elsewhere,
boto3 returns `ResourceNotFoundException` for that name and the
private-key lookup returns the empty string — sync fails the same way
as a missing secret.

Where to find it: AWS console > Secrets Manager > the secret >
"Secret details" panel. The region appears in the URL too:

```
https://<region>.console.aws.amazon.com/secretsmanager/secret?name=<secret-name>
```

Prereqs: The secret must exist in the specified region.

Rotation: n/a. Region cannot change after secret creation. To move
regions, create a new secret in the target region and update both
`GITHUB_APP_PRIVATE_KEY_SECRET_ID` and this value together.

Test vs live: n/a beyond per-environment override.

## GITHUB_APP_PRIVATE_KEY

Purpose: Direct PEM private key — the same string that goes into the
Secrets Manager secret, but stored inline as an
`IntegrationSetting`. Used by
`integrations/services/github_sync/client.py:78` before any AWS
fallback is attempted. Marked as multiline because PEM blocks span
many lines.

Optional in production: the recommended path is the AWS Secrets
Manager indirection above. Leave this field empty in production so
the PEM never lands in the IntegrationSetting table.

Without it: Falls through to the AWS Secrets Manager resolution path.
If that also fails, sync fails as described under
`GITHUB_APP_PRIVATE_KEY_SECRET_ID`.

Where to find it:

- GitHub App settings page > "Private keys" section > "Generate a
  private key". GitHub downloads a `.pem` file. Copy the entire file
  contents (including the `-----BEGIN RSA PRIVATE KEY-----` / `-----END
  RSA PRIVATE KEY-----` lines) into the Studio textarea.
- Direct link, replacing `<your-app>`:

  ```
  https://github.com/settings/apps/<your-app>
  ```

Prereqs: A GitHub App.

Rotation: Safe to rotate. GitHub Apps support multiple active private
keys.

1. GitHub App settings > "Generate a private key" > save the `.pem`.
2. Update this setting via Studio (Integration settings > GitHub App
   > `GITHUB_APP_PRIVATE_KEY`) with the new PEM contents.
3. Once you've confirmed sync works with the new key, return to GitHub
   App settings and delete the old key.
4. Window of impact: zero — both keys are valid until you delete the
   old one.

Test vs live: n/a. Use Secrets Manager for production and leave this
field empty there. For development, pasting the PEM here is fine —
the developer's database is local and ephemeral.
